## For Users

### Requirements

- Overlord CLI (`ovld`) on `PATH`. See the Overlord docs / Settings ‚Üí Agents & MCP.
- Antigravity CLI (`agy`) on `PATH` for Overlord-launched sessions and plugin enablement.
- `MISSION_ID` is exported automatically when an Antigravity session is launched from Overlord.

### Install

From an OpenOverlord checkout:

```bash
ovld agent-setup antigravity
```

This copies the managed plugin files to `~/.gemini/antigravity-cli/plugins/overlord` so `agy` picks it up automatically on next startup (Antigravity crawls its plugin directory and registers skills, hooks, and MCP servers into active memory).

Re-run safely any time the connector contract version changes:

```bash
ovld agent-setup antigravity
ovld doctor
```

### Permission warmup

Antigravity may still prompt once for `ovld protocol` approval on first use. Run this inside an Antigravity session and approve the prefix when prompted:

```bash
ovld protocol help
```

### Namespaced components

Inside Antigravity the plugin surfaces as:

- skills ‚Üí `overlord-mission`, `attach`, `connect`, `load`, `create`, `prompt`, `discuss-objective`, `add-objectives`, `record-work`, `spawn` (each markdown skill also auto-registers as a `/<name>` slash command)
- MCP server ‚Üí `overlord`

Prompts generated by the runner should reference these names in bundle instruction mode.

## For Developers

### Overlord layer map

This connector is intentionally reviewable against the four connector layers in `connectors/README.md`:

- Connector Core: `../../core/overlord-mission/SKILL.md` and its `reference/` files carry the durable attach, update, ask, and deliver workflow.
- Connector Plugins: `skills/overlord-mission/SKILL.md`, `hooks.json`, and `scripts/` extend the core for Antigravity.
- Plugin Adapters: `plugin.json` declares the Antigravity-native plugin package; `mcp_config.json` wires the MCP bridge.
- Prompt Wrappers: `prompt-wrapper.md` describes the launch-time prompt shape for Antigravity.

### What ships

- `plugin.json` ‚Äî Antigravity plugin manifest (name, description, metadata).
- `skills/overlord-mission/SKILL.md` ‚Äî Antigravity adapter template with a `<!-- @connector-core -->` marker; setup interpolates shared core content at install time.
- `skills/attach.md`, `skills/connect.md`, `skills/load.md`, `skills/create.md`, `skills/prompt.md`, `skills/discuss-objective.md`, `skills/add-objectives.md`, `skills/record-work.md`, `skills/spawn.md` ‚Äî one skill per protocol operation; Antigravity auto-registers each as a `/<name>` slash command.
- `hooks.json` + `scripts/pre-invocation-hook.sh` + `scripts/pre-tool-use-hook.sh` ‚Äî follow-up and permission activity capture through `ovld protocol` (see mapping notes below).
- `mcp_config.json` + `scripts/overlord-mcp.mjs` ‚Äî MCP bridge to common `ovld protocol` operations.
- `conformance-manifest.yaml` ‚Äî connector conformance declaration for the Overlord contract.
- `prompt-wrapper.md` ‚Äî Antigravity launch guidance.

The shared Connector Core source lives at `connectors/core/overlord-mission`. `ovld agent-setup antigravity` interpolates that core into the adapter skill template and installs core reference files so the runtime package is self-contained.

### Notes on the Antigravity hook mapping

Antigravity's native hook events are `PreToolUse`, `PostToolUse`, `PreInvocation`, `PostInvocation`, and `Stop` ‚Äî it has no event literally named `UserPromptSubmit` or `PermissionRequest` the way Claude Code and Codex do. This adapter maps Antigravity's native events onto Overlord's canonical hook-type vocabulary the same way the Cursor adapter maps `beforeSubmitPrompt` onto `UserPromptSubmit`:

- `PreInvocation` (fires before each agent turn) ‚Üí recorded as the canonical `UserPromptSubmit` follow-up-capture hook.
- `PreToolUse` (fires before each tool call) ‚Üí recorded as the canonical `PermissionRequest` hook. It always responds `{"allow_tool":true}`; it never gates a tool call, it only logs the request through `ovld protocol permission-request`.

Both hook scripts are defensive: they no-op on any missing `MISSION_ID`, missing `ovld` binary, or unexpected payload shape, and they always return a well-formed allow response so a hook failure never blocks the Antigravity session. Antigravity's public docs do not yet pin an exact `PreInvocation` payload shape the way they do for `PreToolUse` (`toolCall.args`), so `pre-invocation-hook.sh` probes a few plausible field names (`prompt`, `message`, `text`, `input`) for the human turn text; adjust this once the official schema is confirmed.

No `Stop` hook or permission-rule file ships yet ‚Äî Antigravity's pending-delivery check and any declarative auto-approval mechanism should be added once their concrete formats are confirmed, matching the "can be implemented after local MVP" note in `connectors/docs/05-connectors-and-agent-plugins.md`.
