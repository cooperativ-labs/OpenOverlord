# Security Audit - 2026-06-18

## Executive Summary

- Reviewed how Overlord stores and distributes the CLI `USER_TOKEN`: the local `auth.json` credential file and the copy of the same token placed in an agent-pod's environment so a containerized `ovld` instance can authenticate.
- Overall posture is acceptable for current scope. The token is stored as a restricted-permission file rather than world-readable, and the server only ever persists a hash — but the token is long-lived, full-privilege, and unscoped, which increases the impact of any leak.
- No Critical or High findings. The Medium findings (unscoped/non-expiring tokens, duplicated plaintext copies, no sync-folder detection) are the practical levers for reducing blast radius and should be addressed before relying on agent-pods at scale.
- No immediate action is required to block adoption. The recommended next steps are scoped, incremental hardening rather than emergency remediation.

## Product Surface Reviews

### Client-side secrets, tokens, and configuration

#### Finding: USER_TOKEN has no default expiry
- Severity: Medium
- Why it matters: A `USER_TOKEN` confers full account permissions of its creator (`auth/docs/07-user-token-authentication.md:24`) and the spec only reserves `--expires-in` as an *optional* field rather than a default. A token that leaks from a backup, sync folder, or misconfigured log channel remains valid indefinitely until a human notices and runs `ovld user-token revoke`.
- Evidence: `auth/docs/07-user-token-authentication.md:51` ("Optional expiration should be supported or at least reserved as a future field"); no code path in `cli/src/auth-login.ts` or `cli/src/auth-credentials.ts` sets or enforces an expiry on the locally stored credential.
- Recommendation: Default non-interactive/runner-created tokens to a bounded TTL (e.g. 30-90 days) and have `ovld doctor` warn when a stored token is approaching or past a recommended maximum age.

#### Finding: Tokens are unscoped — agent-pod gets full user privilege
- Severity: Medium
- Why it matters: The agent-pod only needs to claim/launch/report on execution requests, but it currently receives the same token type that grants every permission the user has (`auth/docs/07-user-token-authentication.md:20-27`). If the container or its env is compromised, the blast radius is the user's entire account rather than just the runner capability.
- Evidence: `auth/docs/07-user-token-authentication.md:180-199` describes scoped tokens as a *future* extension, not yet implemented; `cli/src/auth-credentials.ts` has a single `AuthCredentialType` (`session_bearer` | `user_token`) with no scope field.
- Recommendation: Prioritize the planned scoped-token extension and mint a `runner`-scoped token specifically for agent-pod use, so a leaked pod credential cannot perform destructive or unrelated account actions.

#### Finding: No detection of cloud-sync paths under the credentials directory
- Severity: Medium
- Why it matters: `auth.json` is written with `0600` permissions (good — see Positive Controls), but nothing checks whether the resolved data directory sits inside a continuously-synced folder (iCloud Drive, Dropbox, OneDrive, Google Drive). If it does, the plaintext token is silently replicated to cloud storage outside the user's awareness, bypassing the local file-permission protection entirely.
- Evidence: `cli/src/auth-credentials.ts:15-17` (`authCredentialsPath()` resolves via `resolveGlobalDataDir()` with no sync-root check); no related check found in the doctor command surface (`cli/test/setup-doctor.test.ts`).
- Recommendation: Add an `ovld doctor` check that flags when the resolved credentials directory is nested inside a known sync-root, and recommend relocating `~/.ovld` or excluding it from sync.

### Authentication and access control

#### Finding: Same raw secret duplicated across two plaintext stores
- Severity: Low
- Why it matters: The user's token exists in both `auth.json` on the host and in the agent-pod's environment. The two copies are not linked for rotation — running `ovld user-token rotate` updates the local file but does not invalidate or refresh a token already injected into a running pod's env, so a compromised pod can continue authenticating with a "rotated away" secret until it's separately restarted with a new value.
- Evidence: `cli/src/auth-credentials.ts:33-56` (`writeStoredAuthCredentials`) only ever writes the local file; no code in this repository propagates rotation to a running container's environment.
- Recommendation: When rotating a token used by an active agent-pod, document/automate that the pod must be restarted with the new token, or move toward short-lived tokens minted per pod-launch instead of reusing the user's standing credential.

#### Finding: Env-var injection is a broader exposure surface than the file
- Severity: Low
- Why it matters: Passing the token via container environment variables (e.g. `docker run -e OVERLORD_USER_TOKEN=...`) makes it visible through `docker inspect`, orchestrator dashboards/logs that capture env at container start, and any same-uid process able to read `/proc/<pid>/environ` — a larger surface than a single `0600` file.
- Evidence: `CLAUDE.md:8` ("In an agent-pod, be sure to use the env variable `OVERLORD_USER_TOKEN`..."); the agent-pod launch mechanism itself is not present in this repository, so the exact injection method could not be directly verified.
- Recommendation: Prefer a `0600`-mode bind-mounted file read at container startup over inline `-e`/`docker-compose environment:` values, matching the protection already used for `auth.json`.

### Logging, telemetry, and error reporting

#### Finding: Token redaction in CLI output appears consistent, but not centrally enforced
- Severity: Informational
- Why it matters: `ovld auth status` (`cli/src/auth-status.ts`) never prints the raw token, and the spec explicitly requires redaction in diagnostics (`auth/docs/07-user-token-authentication.md:176-178`). However, this relies on each call site remembering not to print the token rather than a shared redaction utility, so future code paths (e.g. verbose/debug logging, crash reporters) could regress silently.
- Evidence: `cli/src/auth-status.ts` returns structured status without the token field; no shared redaction helper (e.g. a logger wrapper that masks `out_...` patterns) was found in `cli/src/`.
- Recommendation: Add a small shared redaction utility for anything that might log request/response data, and apply it in debug/verbose logging and error reporting paths as they're added.

## Positive Controls

- `auth.json` is written with explicit `0600` permissions and re-applied via `chmodSync` defensively (`cli/src/auth-credentials.ts:50-55`), restricting access to the owning OS user.
- The backend persists only a secure hash of the token, never the raw secret (`auth/docs/07-user-token-authentication.md:170`); a database compromise does not directly expose usable tokens.
- `ovld auth status` reports credential presence/source without ever revealing the token value (`cli/src/auth-status.ts`).
- `.env*` and `.overlord/tmp/` are already excluded via the repository's `.gitignore`, preventing accidental commits of local secrets and scratch credential files.
- Token lifecycle design (`auth/docs/07-user-token-authentication.md`) already reserves fields for expiration, scoping, and revocation, so the Medium findings above are extensions of an already-planned model rather than a redesign.

## Recommended Next Steps

1. Implement scoped `runner`-only tokens for agent-pod use ahead of other scope types, since this directly bounds the impact of the highest-exposure credential copy.
2. Default a bounded expiry for tokens minted for non-interactive/runner use, and surface token age/expiry in `ovld doctor`.
3. Add an `ovld doctor` check for credentials-directory paths nested inside common cloud-sync roots.
4. When the agent-pod launch mechanism is implemented or reviewed in this repository, re-audit it directly (this report relies on described behavior, not code review, for that surface) and prefer file-mount injection over inline environment variables.
5. Document a rotation procedure that includes restarting/re-provisioning any active agent-pod, until per-launch short-lived tokens remove the need for manual coordination.
