# Security Audit - 2026-07-01

## Executive Summary

- This audit reviews the Overlord backend server surface: the Express API and its
  authentication/authorization model, the `USER_TOKEN` lifecycle, the CLI
  `protocol`/`runner` dispatch, database access patterns, host-side git
  invocation, the upload/serve storage path, and CORS configuration. It broadens
  the prior audit (2026-06-18), which was scoped only to CLI credential storage.
- Overall posture is reasonable and clearly designed. Authentication cleanly
  separates browser sessions, bearer `USER_TOKEN`s, and a loopback fallback, and
  most routes enforce a declarative RBAC permission that is intersected with the
  token's scope. SQL is parameterized, git runs without a shell, uploads are
  served safely, and secrets are kept out of version control.
- No Critical findings. The most important item is the **unconditional
  loopback-trust fallback** for the `protocol`/`runner` API (Medium; escalates to
  High on hosted deployments that terminate or proxy connections on loopback,
  because `trust proxy` is not configured). Two observation routes also bypass
  RBAC/token-scope enforcement, and user-supplied `branchOverride` reaches git
  arguments without the sanitization applied to canonical branch names.
- No issue is an outright blocker for customer evaluation of the local/desktop
  product. Before exposing the backend as a hosted, network-reachable service,
  the loopback-trust fallback and the RBAC gaps below should be remediated.

## Product Surface Reviews

### Authentication and access control

#### Finding: Loopback requests to the protocol/runner API bypass authentication with full operator authority
- Severity: Medium (High on hosted deployments that proxy/terminate on loopback)
- Why it matters: `requireAuthenticatedSession` grants the request the full
  workspace operator identity — with no token and no session — whenever the
  request targets the `protocol`/`runner` surface and originates from a loopback
  address. On the intended local desktop install this is by design, but the
  fallback is **unconditional**: it is not gated by database dialect or a
  hosted/local flag, and `trust proxy` is never set on the Express app (it
  defaults to `false`). If the backend is deployed behind a reverse proxy,
  sidecar, or ingress that connects to the app over `127.0.0.1`/`::1` (a common
  same-host topology), every forwarded request presents a loopback source
  address and is admitted as the operator without credentials — full read/write
  over missions, objectives, sessions, and runner execution requests. Even in a
  purely local install, any other process or local user on the host can drive
  the `protocol`/`runner` API as the operator.
- Evidence: `backend/auth.ts:74-82` (`isLoopbackRequest` uses `req.ip` /
  `req.socket.remoteAddress`); `backend/auth.ts:197-206` (loopback fallback
  resolves `resolveActorForWorkspace` and calls `next()` with no token);
  `backend/index.ts:199-267` (no `app.set('trust proxy', …)` and the fallback
  applies regardless of `DATABASE_DIALECT`).
- Recommendation: Restrict the loopback fallback to the local/desktop profile
  only (e.g. `DATABASE_DIALECT === 'sqlite'` or an explicit
  `OVERLORD_TRUST_LOOPBACK` opt-in), and require a bearer `USER_TOKEN` on the
  protocol/runner surface for hosted (postgres) deployments. Set `trust proxy`
  deliberately so `req.ip` reflects the real client and cannot be spoofed to
  loopback by an intermediary. Document the local-trust assumption explicitly.

#### Finding: USER_TOKEN hashes are unsalted SHA-256 without a server-side pepper
- Severity: Informational
- Why it matters: Tokens are stored as a bare `sha256(rawToken)` hash. Because
  raw tokens carry 256 bits of entropy (`randomBytes(32)`), offline brute force
  is infeasible and the choice is acceptable. An HMAC keyed with a server-side
  secret (pepper) would add defense-in-depth so that a database-only compromise
  yields nothing usable even against low-entropy future token formats.
- Evidence: `auth/src/auth/token.ts:33-35` (`hashToken` = plain SHA-256);
  `auth/src/auth/token.ts:37-39` (256-bit random token body).
- Recommendation: Consider HMAC-SHA256 with a secret loaded from the environment
  (not the database) if token storage hardening is later prioritized.

### API routes and server actions

#### Finding: Execution-target observation routes enforce no RBAC permission or token scope
- Severity: Medium
- Why it matters: `POST /api/execution-targets/:id/observations` and
  `POST /api/execution-targets/:id/mission-branch-observations` are registered
  with only `{ mutates: true }` and no `requires`. Every other mutating route
  passes a `requires` permission through `handle()`, which calls
  `requirePermission` — the single place that intersects the actor's role grants
  with the authenticating token's scope. Because these two routes skip that gate,
  a deliberately narrow `USER_TOKEN` (e.g. one scoped to `mission:read`) can
  still post observations that mutate mission branch/execution-target state. The
  token-scope restriction is silently not enforced for this surface.
- Evidence: `backend/index.ts:1025-1046` (both routes use `{ mutates: true }`
  with no `requires`); `backend/index.ts:245-261` (`handle` only enforces scope
  when `options.requires` is set); `backend/rbac.ts:61-89` (`requirePermission`
  is the sole scope-intersection point).
- Recommendation: Add an explicit `requires` (e.g.
  `PERMISSIONS.EXECUTION_REQUEST_CLAIM` or a dedicated observation permission)
  to both routes so scoped tokens and under-privileged actors are constrained
  consistently with the rest of the API.

#### Finding: Stored objects are authorized at workspace scope only, not per project/mission
- Severity: Low
- Why it matters: `GET /api/storage/:bucketKey/:storageKey` gates on a blanket
  `PROJECT_READ` permission, and `resolveStoredObject` looks the object up by
  `(storage_bucket_id, storage_key)` scoped only to the active workspace — it
  does not verify that the caller may read the specific project/mission the
  attachment belongs to. Any actor holding `PROJECT_READ` who learns an
  attachment's storage key can fetch it regardless of which project it belongs
  to. Impact is bounded by unguessable UUID storage keys and the current
  single-workspace, single-operator trust model, but the authorization is
  coarser than the per-project isolation the data model expresses.
- Evidence: `backend/index.ts:488-516` (route gate is `PROJECT_READ`);
  `backend/storage.ts:520-536` (`resolveStoredObject` filters by bucket + key +
  workspace, with no project/mission authorization).
- Recommendation: When multi-project or multi-tenant isolation becomes a
  requirement, scope object reads to the caller's authorized project/mission
  (join through the attachment's `project_id`/`mission_id`) rather than relying
  on key unguessability.

### Edge functions, background jobs, and integrations (host-side git)

#### Finding: User-supplied `branchOverride` reaches git arguments without branch-name sanitization
- Severity: Low
- Why it matters: `updateMission` persists `body.branchOverride` verbatim into
  `missions.branch_override`, whereas canonical branch names are always passed
  through `sanitizeBranchName`, which strips leading `-`/`.`, `..`, and other
  ref-hostile characters. Branch names flow into host-side git operations
  (`git merge <branch>`, `git push origin <branch>`, `git push -u origin
  <branch>`) as **positional arguments with no `--` end-of-options separator**.
  A crafted override beginning with `-` could therefore be interpreted by git as
  an option rather than a ref (argument injection), and an override containing
  path-like segments could steer worktree/ref resolution. Exploitation requires
  an authenticated `MISSION_UPDATE` actor and a local backend co-located with the
  checkout, so real-world impact is limited, but the input is unsanitized on a
  path the canonical flow deliberately hardens.
- Evidence: `backend/repository.ts:3075-3079` (`branchOverride` written without
  `sanitizeBranchName`); `backend/branch-planning.ts:73-89` (sanitizer that
  canonical names use, stripping leading `-`/`.`); git arg arrays without `--`
  in `packages/core/service/local-target/branch-actions-git.ts:49,80-87,159,180`.
- Recommendation: Run `branchOverride` through `sanitizeBranchName` on write, and
  add a `--` end-of-options separator before branch/ref positionals in the git
  argument arrays as defense-in-depth.

### Client-side secrets, tokens, and configuration

#### Finding: CORS wildcard patterns expand `*` to `.*` under credentialed CORS
- Severity: Informational
- Why it matters: `matchesOriginPattern` converts a `*` in an allowed-origin
  entry to `.*`, which matches across dot boundaries, so an operator-supplied
  wildcard in `OVERLORD_WEB_ORIGINS` is broader than a single-subdomain pattern
  suggests. Combined with the app's `credentials: true` CORS, an overly-permissive
  operator wildcard could admit credentialed requests from unintended origins.
  The default allowlist is loopback-only and safe; this only affects operators
  who configure wildcard origins.
- Evidence: `backend/browser-origins.ts:30-34` (`*` → `.*`);
  `backend/index.ts:201-225` (`credentials: true` with origin reflection against
  the allowlist).
- Recommendation: Expand `*` to a single-label matcher (`[^.]*`) or document the
  wildcard's breadth so operators understand the exposure before using it.

## Positive Controls

- Declarative RBAC: nearly every route declares a `requires` permission enforced
  by `requirePermission`, which intersects the actor's role grants with the
  authenticating token's scope before the handler runs
  (`backend/index.ts:245-261`, `backend/rbac.ts:61-89`). Protocol subcommands are
  gated the same way via `SUBCOMMAND_PERMISSIONS` (`backend/protocol.ts:457-501`).
- `USER_TOKEN` hardening: 256-bit random secret, hash-only storage, a display
  prefix instead of the raw value, and a **90-day default TTL** — the prior
  audit's expiry recommendation is now implemented (`auth/src/auth/token.ts:17,
  79-118, 230-277`).
- SQL is fully parameterized; the few dynamic query fragments interpolate only
  fixed column/field constants (`selectMissionsSql`, `USER_TOKEN_COLUMNS`, the
  hardcoded `fields` array in `heartbeatSession`), never user input.
- Host-side git runs via `execFileSync('git', args)` with no shell, eliminating
  shell-injection (`packages/core/service/local-target/git-run.ts`).
- Upload serving resolves the exact recorded backend key before streaming, so
  `storageKey` path traversal is impossible; SVG uploads are rejected and
  attachments are served with `X-Content-Type-Options: nosniff` and a download
  disposition to prevent inline script execution (`backend/storage.ts:54-64,
  514-582`, `backend/index.ts:488-516`).
- Secrets hygiene: `.env.prod` / `.env.local` are gitignored and untracked; only
  `*.example` files are committed. Trust-proxy defaults keep `req.ip` bound to the
  real socket peer.
- Auth surfaces are cleanly separated: browser Better-Auth sessions, bearer
  `USER_TOKEN`, and the loopback fallback are distinct branches, and browser
  `/api` routes deliberately never receive the loopback fallback
  (`backend/auth.ts:156-213`).

## Recommended Next Steps

1. Gate the loopback-trust fallback to the local/desktop profile (or an explicit
   opt-in), require a `USER_TOKEN` on the protocol/runner surface for hosted
   deployments, and set `trust proxy` deliberately before any network-reachable
   deployment.
2. Add `requires` permissions to the two execution-target observation routes so
   token scope is enforced uniformly.
3. Sanitize `branchOverride` on write and add `--` end-of-options guards to the
   git argument arrays.
4. Scope stored-object reads to the caller's authorized project/mission if/when
   multi-project isolation is required.
5. Optional hardening: HMAC token hashing with a server pepper, and tighten CORS
   wildcard expansion to single-label matching.
