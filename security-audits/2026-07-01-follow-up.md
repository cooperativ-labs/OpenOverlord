## Product Surface Reviews

### Authentication and access control

No confirmed findings in the reviewed invitation and member-management paths.

Evidence:
- Invitation-management routes are permission-gated:
  `GET/POST/DELETE /api/workspaces/:id/invitations` require
  `INVITATION_READ`, `MEMBER_INVITE`, and `INVITATION_REVOKE`
  respectively, while member removal and role changes require
  `MEMBER_REMOVE` and `ROLE_ASSIGN`
  (`backend/index.ts:417-452`).
- The service layer adds an ADMIN check before invite, revoke, role-change, and
  member-removal operations (`backend/workspaces.ts:879-907`,
  `backend/workspaces.ts:1013-1046`, `backend/workspaces.ts:1238-1297`,
  `backend/workspaces.ts:1302-1356`).
- Cross-workspace resolution is explicitly rejected with `403`, and concurrent
  tenant isolation is covered in the conformance suite
  (`backend/workspace-isolation.postgres-conformance.test.ts:189-217`,
  `backend/workspace-isolation.postgres-conformance.test.ts:121-183`).

### Invitation tokens and acceptance flow

No confirmed findings in token generation, storage, or acceptance semantics.

Evidence:
- Invitation secrets are generated from 24 random bytes plus an `inv_` prefix,
  then stored only as a SHA-256 hash; the raw token is returned once and never
  persisted (`backend/workspaces.ts:831-840`, `backend/workspaces.ts:941-983`).
- Acceptance resolves by `token_hash`, requires an authenticated profile, and
  enforces exact lowercased email match against the invited address before
  creating membership (`backend/workspaces.ts:1056-1116`).
- Pending invitations become single-use on acceptance and are rejected after
  revoke or expiry; expiry now persists as a committed `expired` state rather
  than rolling back (`backend/workspaces.ts:1067-1102`,
  `backend/workspace-invitations.test.ts:143-208`).

### IDOR and workspace-scoped route access

No confirmed insecure direct object reference was found on the reviewed
workspace-member and invitation routes.

Evidence:
- The reviewed `/api/workspaces/:id/*` routes sit behind permission checks in
  `handle(...)`, and the mutating invitation/member services also verify the
  caller is an admin of the target workspace (`backend/index.ts:417-466`,
  `backend/workspaces.ts:246-252`).
- My Missions isolation now has direct regression coverage proving one tenant's
  mission list excludes another tenant's entries
  (`backend/my-missions.test.ts`).

## Positive Controls

- Invitation tokens have high entropy and are stored hash-only.
- Acceptance is authenticated and email-bound, which prevents a valid token from
  being redeemed by the wrong signed-in account.
- Re-inviting supersedes an existing pending invite instead of creating multiple
  concurrently valid invites for the same workspace/email pair.
- Member promotion/demotion and removal enforce a last-admin guard.
- The regression suite now covers happy path, fabricated token rejection,
  revoked token rejection, expired token rejection, tenant-isolated My Missions
  reads, and SQLite request-context isolation.

## Recommended Next Steps

1. Run the same Phase 5 battery with `TEST_DATABASE_URL` configured so the
   PostgreSQL conformance path is exercised, not just present.
2. Keep the invitation acceptance path under regression coverage if route or
   auth middleware changes, because expiry/state-transition handling is easy to
   break when exceptions occur inside transactions.
