# Native Node Modules Isolation Plan

**Status:** Proposal (planning only)
**Contract baseline:** `0.18-draft`
**Problem:** The same checkout is used by macOS host tooling, Linux agent-pod containers, and Electron desktop packaging. Native Node addons, especially `better-sqlite3`, are currently resolved from shared `node_modules` paths. That makes the active install depend on whichever runtime last installed or rebuilt the addon.

## 1. Diagnosis

`better-sqlite3` ships a native `.node` binary. That binary must match all of:

- Operating system binary format: macOS `Mach-O` vs Linux `ELF`
- CPU architecture: `arm64` vs `x64`
- Node module ABI: system Node vs Electron's embedded Node

We confirmed agent-pod runs:

```text
linux arm64 modules=137
```

while the shared root addon visible inside agent-pod starts with:

```text
cf fa ed fe
```

That magic byte sequence is macOS Mach-O. A Linux process loading it fails with `invalid ELF header`.

There is a separate failure class where the OS matches but the Node ABI does not:

```text
NODE_MODULE_VERSION 146 vs 147
```

That means the addon was compiled for a different Node/Electron ABI. It is not fixed by changing OS isolation alone.

## 2. Target Model

Each native runtime gets its own install tree:

| Runtime | Platform | Node ABI owner | `better-sqlite3` location |
| --- | --- | --- | --- |
| macOS host dev/tests | `darwin arm64` | `.nvmrc` / shell Node 24 | repo root `node_modules` |
| Linux agent-pod | `linux arm64` | container Node | container-owned `node_modules` volume |
| Packaged Desktop | `darwin arm64` / future `darwin x64` | Electron embedded Node | staged `desktop/node_modules/better-sqlite3` |

The root `node_modules` directory must never be shared between host and Linux containers. Electron packaging must never mutate the root `better-sqlite3` addon.

## 3. Implementation Plan

### Phase 1: Agent-pod `node_modules` isolation

Configure agent-pod so the repo source is mounted into the container, but `node_modules` is shadowed by a Linux-owned volume.

Required isolated paths:

```text
/Users/jake/Development/Cooperativ/OpenOverlord/node_modules
```

Recommended additional isolated paths if they exist or are generated by tooling:

```text
/Users/jake/Development/Cooperativ/OpenOverlord/webapp/node_modules
/Users/jake/Development/Cooperativ/OpenOverlord/desktop/node_modules
/Users/jake/Development/Cooperativ/OpenOverlord/database/node_modules
/Users/jake/Development/Cooperativ/OpenOverlord/cli/node_modules
/Users/jake/Development/Cooperativ/OpenOverlord/auth/node_modules
/Users/jake/Development/Cooperativ/OpenOverlord/automations/node_modules
```

After changing mounts, run the install inside agent-pod:

```bash
yarn install
```

Acceptance checks inside agent-pod:

```bash
node -p "process.platform + ' ' + process.arch + ' modules=' + process.versions.modules"
node -e "const fs=require('fs'); const b=fs.readFileSync('node_modules/better-sqlite3/build/Release/better_sqlite3.node').subarray(0,4); console.log([...b].map(x=>x.toString(16).padStart(2,'0')).join(' '))"
node -e "const Database=require('better-sqlite3'); const db=new Database(':memory:'); console.log(db.prepare('select 1 ok').get().ok); db.close()"
```

Expected binary magic inside Linux:

```text
7f 45 4c 46
```

### Phase 2: Host Node version discipline

Keep the repo's host development runtime on Node 24, matching `.nvmrc`.

Acceptance checks on macOS host:

```bash
node -v
which node
node -p "process.platform + ' ' + process.arch + ' modules=' + process.versions.modules"
node -e "const Database=require('better-sqlite3'); const db=new Database(':memory:'); console.log(db.prepare('select 1 ok').get().ok); db.close()"
```

The shell should resolve NVM's Node 24 before `/opt/homebrew/bin/node`. If agent tooling launches with a different `PATH`, fix the launcher environment rather than changing the project to Node 26.

### Phase 3: Desktop packaging isolation

Keep Electron packaging's native rebuild explicit and local to `desktop/node_modules/better-sqlite3`.

Planned code changes:

1. Set `npmRebuild: false` in `desktop/electron-builder.yml`.
2. Keep `scripts/build-desktop.ts` `stageNativeAddon()` as the only Electron ABI rebuild path.
3. Add a post-package guard that verifies:
   - staged desktop addon loads under Electron with `ELECTRON_RUN_AS_NODE=1`
   - root addon still opens an in-memory SQLite DB under host Node

Acceptance checks after `yarn desktop:package`:

```bash
node -e "const Database=require('better-sqlite3'); const db=new Database(':memory:'); console.log(db.prepare('select 1 ok').get().ok); db.close()"
ELECTRON_RUN_AS_NODE=1 node_modules/electron/dist/Electron.app/Contents/MacOS/Electron -e "const Database=require('./desktop/node_modules/better-sqlite3'); const db=new Database(':memory:'); console.log(db.prepare('select 1 ok').get().ok); db.close()"
```

### Phase 4: Automated preflight

Add a diagnostic script, for example `scripts/check-native-runtime.ts`, that prints and validates:

- Host `process.platform`, `process.arch`, `process.versions.modules`
- `node` path
- Root `better_sqlite3.node` magic bytes
- Root in-memory SQLite load result
- Electron runtime ABI when Electron is installed
- Desktop staged addon load result when present

This gives agents a cheap first check before running webapp/database tests.

## 4. Operational Rules

- Do not sync or share `node_modules` across macOS and Linux.
- Do not run `electron-rebuild` against the workspace root.
- Do not run `npm rebuild better-sqlite3` inside a container against the host-mounted root `node_modules`.
- Re-run `yarn install` after changing the runtime that owns an isolated install tree.
- Treat `invalid ELF header` as an OS-format mismatch, not a Node-version mismatch.
- Treat `NODE_MODULE_VERSION` errors as an ABI mismatch, not an OS-format mismatch.

## 5. Contract Impact

No contract change is required. This plan changes development and packaging hygiene only. It does not add a component, alter an interaction surface, change a protocol command, or modify database schema.
